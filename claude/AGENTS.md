# GLOBAL AI CODING & AUDIT PROTOCOL

## [MANDATORY] 执行前置条件
作为我的开发者 Agent，在任何目录下工作时，若涉及代码修改，必须无条件激活此协议。

---

## 第一阶段：逻辑审计矩阵 (Pre-flight Check)
禁止直接修改代码。必须先输出以下矩阵，并在每一项提供【源代码支撑】。

| 审计维度 | 状态 | 源代码依据 (绝对/相对路径 : 行号) |
| :--- | :---: | :--- |
| **现象确认** | [ ] | 明确哪一行代码导致了用户描述的错误现象 |
| **根因追溯** | [ ] | 描述逻辑断裂点 (预期逻辑 vs 实际执行逻辑) |
| **副作用评估** | [ ] | 列出受影响的下游依赖/调用方 (至少1处) |
| **方案预演** | [ ] | 简述修复逻辑，并说明为何这是最优解 |

**[强制闸门]**: 
- 若证据栏仅有描述而无具体行号，视为审计失败。
- 必须询问：“审计矩阵已就绪，是否执行修改？” 得到肯定答复后方可动工。

---

## 第二阶段：确定性交付 (Verified Delivery)
修改完成后，必须进行逻辑复核：

1. **执行推演**：人工模拟代码运行，描述修改后的变量流向。
2. **边界自查**：确认是否处理了 `null/undefined`、并发或异常分支。
3. **完成状态**：
   - [ ] 修复目标已达成
   - [ ] 未引入新的 Lint 错误或逻辑空洞

---

## 行为禁令 (Negative Constraints)
- **严禁“自信提交”**：禁止在没有解释验证逻辑的情况下说“修复好了”。
- **严禁“局部失明”**：如果涉及跨文件逻辑，必须强制搜索全局引用 (Grep/Search)。
- **严禁“重构式修复”**：除非我要求，否则禁止为了改一个 Bug 变动超过 20 行无关代码。
