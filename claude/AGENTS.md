# 🤖 核心逻辑准则 (Karpathy 版本)

## 1. 动手前：分级审计矩阵

判断任务等级：[Level 1: 纯文案/UI] 直接修；[Level 2: 逻辑/跨文件] 必填表。

| 审计维度        | 状态 | 逻辑证据 (文件:行号 & 代码片段)               |
| :-------------- | :--: | :-------------------------------------------- |
| **触发条件**    | [ ]  | 明确是 Bug/Feature/Refactor 中的哪一种        |
| **根因/触发点** | [ ]  | 明确 Expected vs Actual                       |
| **全量副作用**  | [ ]  | **受影响范围 > 1 处时必须全部列出下游调用方**  |

> **强制闸门**：
> - Level 1：直接修，修完请用户验证并给出验证方案
> - Level 2：填表后询问”是否修改？”。若用户拒绝，立即停止并请求纠偏，严禁盲目重试。

## 2. 行为禁令 (Negative Constraints)

- **Surgical Changes**: 仅做外科手术式修改，严禁改动 > 20 行无关代码或格式。
- **Simplicity First**: 严禁为了“灵活性”增加多余抽象，用最简单的代码解决问题。
- **Stop when Confused**: 严禁猜代码。有多种理解或逻辑疑问时，必须先提问。
- **Goal-Driven**: 修改后必须推演数据流，确保不仅“修好了”，且“没改坏”。

## 3. 提交要求

- 必须清理因本次修改产生的孤儿代码（imports/vars）。
- 禁止在未说明验证逻辑的情况下说“修复好了”。
